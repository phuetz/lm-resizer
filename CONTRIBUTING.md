# Contributing

This project is a Rust-only port. Keep contributions aligned with that boundary:
no Python runtime surface, package manifest, helper script, or required Python
tooling should be added.

## Local Verification

Before opening a change, run the release check for your platform:

```bash
./scripts/check-release.sh
```

Windows:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts\check-release.ps1
```

For release packaging changes, also run:

```bash
./scripts/package-release.sh
```

Windows:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts\package-release.ps1
```

## Custom Command Filters

Use project-local filters when a command family is noisy but too specific for a
built-in rule:

```bash
lm-resizer init-filters --profile generic --path .lm-resizer/filters.toml
lm-resizer verify-filters --path .lm-resizer/filters.toml --json
lm-resizer audit-filters --path .lm-resizer/filters.toml
lm-resizer audit-filters --path .lm-resizer/filters.toml --review
lm-resizer trust-filters --path .lm-resizer/filters.toml
```

Filter pull requests should include:

- a representative command output fixture or inline test;
- a `verify-filters` pass;
- an `audit-filters --review` report when the filter file is new or changed;
- a short explanation of what gets removed and what must remain visible.

Prefer adding a project filter first. Promote a filter to a built-in only when
the command pattern is reusable across projects.

## Provider Fixtures

Provider fixtures must be safe to commit. Start from a real payload, sanitize it,
then review the output before opening a pull request:

```bash
lm-resizer sanitize-provider-fixture \
  --provider openai \
  --input payload.json \
  --output fixtures/provider-cache/openai-real-case.json
```

Use the matching provider value: `openai`, `anthropic`, `bedrock`, or `vertex`.
Do not commit real secrets, bearer tokens, account identifiers, customer text,
or unsanitized large prompts.

## Agent Integrations

Agent integration changes should stay opt-in and reversible:

- generated files belong under project-local config such as `.codex/`,
  `.claude/`, or `.lm-resizer/`;
- installers must use marked blocks when editing agent instruction files;
- unknown hook event shapes must not fail the agent command path;
- tests should cover both the generated config and the runtime parser.

## Release Changes

Release-related pull requests should update these files together when relevant:

- `Cargo.toml`
- `crates/lm-resizer-core/Cargo.toml`
- `crates/lm-resizer-wasm/Cargo.toml`
- `packages/wasm/package.json`
- `docs/RELEASE.md`
- `dist/release-evidence.json` generated by the release check
- `dist/SHA256SUMS` generated by the release/package scripts

Real npm publication is intentionally separate from normal CI. Use the protected
`Publish WASM Package` workflow only after the release checklist is approved.
